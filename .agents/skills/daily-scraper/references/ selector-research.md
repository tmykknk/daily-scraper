# セレクタ調査の詳細手順

`scrape.py` の TODO セレクタを埋める、または壊れたセレクタを直す際に参照する。

## なぜ「TOP10全体を囲む1つのセレクタ」が存在しないことがあるのか

楽天のランキングページは、上位TOP3とそれ以降(4位以降)を別レイアウトのリストで表示する構成になっている。

```
<div class="rnkRanking_top3box"> ... 1〜3位のカード ... </div>
<div class="rnkRanking_after4box"> ... 4位以降のリスト項目 ... </div>
```

このケースでは `TODO_TOP10_AREA_SELECTOR` として単一の親要素を指定できない。
以下のいずれかの方針で対応する。

### 方針A: 共通の祖先要素を使う
2つのエリアを両方内包する、より外側の共通コンテナがあればそれを使う
(見た目上は無駄な余白が入るが、実装はシンプル)。

### 方針B: 個別エリアのスクショを合成する
Playwright/Pillowで「TOP3エリア」と「4〜10位エリア」を別々にスクリーンショットし、
Pillowで縦に結合してから1枚のWebPとして保存する。
`scrape.py` の `scrape_top10()` はこの場合、複数エリアのパスを受け取って結合する処理に変更する。

### 方針C: 個別アイテムのバウンディングボックスから合成領域を計算する
1位〜10位の各アイテム要素の `bounding_box()` を取得し、それらを内包する最小の矩形を計算して、
`page.screenshot(clip=...)` で一括スクショする。レイアウトが不規則な場合に有効。

どの方針を取るかは実際のDOM構造を見てから判断すること(推測で決め打ちしない)。

## Playwright MCPでの調査手順(具体例)

1. Playwright MCPでページを開く
   - URL: `https://ranking.rakuten.co.jp/daily/558885/`
2. ページのスナップショットを取得し、全体構造をざっと把握する
3. 「ランキング」「1位」「順位」などのテキストを手がかりに、繰り返し構造(商品アイテム)を探す
4. 見つけた候補要素に対して、以下を確認する:
   - class名やdata属性が安定していそうか(ハッシュ化された動的クラス名でないか)
   - 同じ構造が10件分(またはそれ以上)存在するか
   - 商品名・価格・リンクがその要素の子孫として取得できるか
5. TOP3とTOP4〜10で構造が違う場合は、上記の方針A/B/Cのどれが適切か判断する
6. 候補セレクタが決まったら、必ず実際に `locator(...).count()` で件数を確認し、
   `inner_text()` / `get_attribute("href")` で値が正しく取れるか確認してから `scrape.py` に反映する

## セレクタの安定性についての注意

- 動的に生成されるクラス名(例: `css-a1b2c3` のような形式)は避け、
  意味のあるクラス名・data属性・要素の階層構造を優先する
- 迷ったら `:has-text()` やテキストベースのフィルタと組み合わせて安定性を上げる方法もある
- 変更を加えたら、日を変えて(できれば別の時間帯に)再実行し、
  ページの動的な要素(広告枠の有無など)でセレクタが崩れないか確認するとより安全
